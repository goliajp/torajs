# Phase review — post M4 + M6.1 + M6.2 (2026-05-01)

After M0-M4 + M6.1 + M6.2 + closure / generic / array-method fast-paths shipped, we ran:

1. A code review by an Explore agent reading the 5 main files.
2. Manual edge-case probing (unicode, overflow, OOB, nested closure, throw-inside-finally, mutual recursion, mutable capture).
3. The full bench harness against rust / go / bun-jsc / bun-aot / node-v8 / python.

The benches are clean (15 / 15 green on JIT + AOT), but the edge probes found six **correctness bugs** the existing tests don't exercise, plus several design smells. The bench picture is also competitive — torajs trails rust on 4 cases by less than 25% and beats it on others, so the "compete with bun / rust" thesis is intact.

This doc is the finding list + fix priorities. Bugs marked **P0** break correctness on supported features. **P1** misbehaves silently on some shape. **P2** is code-quality / maintainability.

## Bugs (in priority order)

### P0 — `return` inside `try` / `catch` skips `finally`

```ts
function f(): number {
  try { return 1; } catch (e) { return 99; } finally { console.log("finally A"); }
}
console.log(f());   // expected: "finally A\n1"; actual: "1"
```

The `Stmt::Return` lowerer currently emits `Terminator::Ret` immediately, jumping past any active `finally`. TS spec requires `finally` to run on every exit path, including `return`. Resource cleanup placed in `finally` is silently skipped.

**Fix sketch:** add a per-fn "pending return slot" + a flag bit. When `return v` executes inside a try-with-finally, store v + set flag + branch to the innermost finally. At finally tail, branch logic becomes: `pending_return` → `load + ret`; else `throw_active` → propagate; else fall-through to `after_blk`. The current finally tail already handles the throw_active branch — we add the pending_return branch in front.

Severity: P0. Documented as "known limitation" in commit `53762d0`'s message but not in user-facing docs, and most users will hit it without noticing.

---

### P0 — `throw` inside `finally` lost when `try` had `return`

```ts
function f(): number {
  try { return 1; } catch (e) { return 2; } finally { throw 99; }
}
function g(): number {
  try { return f(); } catch (e) { return 100 + e; }
}
console.log(g());   // expected: 199; actual: 1
```

Same root cause as the previous bug — the `return` short-circuits past the finally's `throw`. Fixing #1 fixes this for free.

---

### P0 — Caught non-Copy value leaks on `catch` fall-through

```ts
function tryIt(): number {
  try { throw "msg"; } catch (e: string) { return 1; }
}
for (let i = 0; i < 1000; i = i + 1) tryIt();
// leaks --atExit reports 999 leaks (~95 KB)
```

`Stmt::Try` lowering marks the catch param `moved: !is_copy()` (`ssa_lower.rs:2118`) so the fn-end drop walk skips it. But there's no per-block drop emitted on catch fall-through either, so the heap behind `e` is never freed. Throw-handling code in real programs that catches by value and falls through (vs re-throwing) leaks every time.

**Fix:** emit drops for the catch param and any locals declared in the catch body before the `Br(post_target)` that closes the catch block.

---

### P0 — Nested capturing closures crash at SSA lower

```ts
function outer(a: number): number {
  let inner = (b: number): number => {
    let inner2 = (c: number): number => a + b + c;
    return inner2(100);
  };
  return inner(10);
}
// not yet supported: ssa-lower: lifted closure `__closure_0` has no capture types
```

`lift_arrow_fns` walks expressions low→high, so the inner arrow becomes `__closure_0` and the outer becomes `__closure_1`. Both get appended to `ast.stmts` in that order. Pass-2 of `ssa_lower` lowers in append order: `__closure_0` is lowered before `__closure_1`, but `__closure_0`'s captures table is populated by the **outer** closure's body lowering (which constructs `__closure_0`).

**Fix:** lower lifted closure FnDecls in **reverse append order** (outermost first) so each capturing closure's construction site lowers before its body. Or: thread `closure_captures` as `Cell<HashMap<...>>` so it's accumulated in any order, then resolved at body-lower time.

---

### P1 — `try { } finally { }` (no catch) parse error

```ts
try { ... } finally { ... }   // parse error: expected `catch`
```

Parser at `parser.rs:486` requires a `catch` clause. TS allows `try { } finally { }` without catch. Single-line fix.

---

### P1 — `xs[i]` on empty / OOB array returns garbage, no bounds check

```ts
let xs: number[] = [];
console.log(xs[0]);   // outputs `2` (random heap value); expected: trap or undefined
```

The `__torajs_arr_alloc(0)` path allocates header bytes (16) and zero element capacity. Reading `xs[0]` does `LoadDyn(elem_ty, ptr, 0*8 + 16)` — reads 8 bytes past the header, which is uninitialized heap. No bounds check at index time.

**Fix options:**
- Cheap: emit `if (i >= len) panic` before the LoadDyn. ~3 instructions per read; some perf cost on hot loops.
- Lazy: trust the user (current behavior). Document.
- Optimize-aware: emit the check, let LLVM/Cranelift hoist or remove if it can prove i < len. Probably the right answer long-term.

---

### P1 — Generic fn applied to struct type panics

```ts
type A = { v: number };
function id<T>(x: T): T { return x; }
let a: A = { v: 5 };
let a2: A = id(a);   // panic at lower: "void cannot be a parameter type"
```

`check::type_to_ann` returns `"void"` for `Type::Struct(_)` (with a TODO comment). The mono cache then sees param type `void` and lowers a fn with `void` parameters, which both backends reject.

**Fix:** embed structural identity in the annotation string (e.g. `__struct(<field_anns_joined>)`) so each unique struct produces a unique annotation; then `parse_type` decodes it back to a registered `Type::Obj(StructId)`.

---

### P2 — Mutable closure captures are by-value, not by-reference

```ts
function makeCounter(): number {
  let n: number = 0;
  let inc = (): number => { n = n + 1; return n; };
  return inc() + inc() + inc();   // expected (TS): 1+2+3=6; actual: 1+1+1=3
}
```

This is documented as "out of scope for M2.A" (the env stores captures by value), but it's not flagged in `docs/ts-subset.md`. **Action:** add to the rejected-features list.

## Design smells (from agent + manual review)

These don't break anything yet but will compound as scope grows.

1. **Capture names smuggled via type annotation string** (`ast.rs:236`) — `__env(cap0|cap1|...)` is parsed and reparsed across pass boundaries. Fix: extend `Param` with `captures: Option<Vec<String>>`.
2. **Generic mono cache keyed by ann strings** (`ssa_lower.rs:175-188`) — caches by `Vec<String>` (output of `type_to_ann`). Two distinct types can collide if their type-to-ann is identical (the struct case above is the active example). Fix: key by `Vec<check::Type>` directly.
3. **`is_string_borrow` allow-list duplicated across check.rs and ssa_lower.rs** — the borrow-method names live in two places. Fix: export a `pub const STRING_BORROW_METHODS: &[&str]` from `check.rs` and import.
4. **40+ panics in `ssa_lower.rs`** — e.g. `parse_type` panics on unknown name (line 1153). These should be `Result<_, LowerError>` so user-facing `tr build` doesn't abort the whole process.
5. **No integration test for the C runtime link path** — `runtime_str.c` is compiled+linked per `tr build`. If the symbol names drift between SSA declarations and the C source, AOT will fail at link, but JIT will keep working — silent backend divergence. Fix: add a smoke test for split / join under both backends.

### Compile time regression

| stage | median compile | binary |
|---|--:|--:|
| pre-runtime_str.c (Apr 30) | ~50 ms | 33.9 KB |
| post-runtime_str.c (this) | ~91 ms | 35.8 KB |

The added 40 ms is `cc -c -O2 runtime_str.c` per `tr build`. Two paths to recover:
- Cache the compiled `.o` once per `tr` process (OnceLock<PathBuf>) — saves ~40 ms on every build after the first within the same process.
- Pre-build the runtime as a static lib at `cargo build` time and embed via `include_bytes!` — saves ~40 ms on every build.

## Recommended fix order

1. Nested closure lowering (P0, isolated, ~30 LOC fix) ← cleanest win
2. `try { } finally { }` parser (P1, 5 LOC) ← also cleanest
3. Catch param drop (P0, ~20 LOC) ← stops the leak
4. `type_to_ann` for structs (P1, ~30 LOC) ← unblocks generic + struct combos
5. `return + finally` (P0, ~80 LOC, design needed) ← biggest, pending-return slot
6. Compile-time regression (perf — cache C runtime .o)
7. Doc: update `docs/ts-subset.md` with mutable-capture caveat

Items 1–4 are short and pay for themselves. #5 needs design discussion.
